# Log Viewer Module Hardening

## Overview

This PRD refines the Log Viewer requirement into a hardening pass for the existing IdeaDump `/log-viewer` tool.

The repository already includes:

- `app/log-viewer/page.tsx`
- `components/organisms/LogViewer.tsx`
- `lib/logViewer/parse.ts`
- `lib/logViewer/transactions.ts`
- `lib/logViewer/types.ts`

The goal is not to build a brand-new module from scratch. The goal is to complete the existing client-side debugging tool so raw API logs can be pasted or uploaded, parsed into structured request/response calls, searched, filtered, and inspected without manual scanning.

---

## Problem Statement

API logs are emitted as raw, irregular text lines that mix timestamps, request markers, response markers, URLs, HTTP status codes, and inline JSON payloads. A single API call can be split across several lines with no reliable shared ID. Type labels are inconsistent, including examples such as:

- `REQUEST`
- `RESPONSE_SUCCESS`
- `RESPONSE ERROR3`
- `AUTH_REQUEST`
- `CONTENT DATA >>`

Large nested JSON bodies are printed inline, and important debugging context may appear in a `CONTENT DATA` line before the actual request line.

This makes debugging slow because the current workflow requires visually scanning raw text to find a failing call, match it to the originating request, and inspect its payload.

---

## Goals

- Harden the existing `/log-viewer` route as a client-only debugging tool.
- Support both pasted raw logs and uploaded `.log` / `.txt` files.
- Parse raw lines into structured log events while preserving original line numbers.
- Pair request and response lines into logical API calls.
- Highlight failed calls and orphaned request/response lines.
- Prioritize search and filtering as the primary workflow.
- Show unparsed lines separately so no log content disappears silently.
- Keep all raw log processing local to the browser for v1.

---

## Non-Goals

- Persisting uploaded or pasted logs to Supabase.
- Persisting parsed logs to browser storage.
- Adding a new RBAC module slug.
- Adding a database migration.
- Adding a third-party JSON viewer library.
- Building a chronological timeline view.
- Editing, annotating, or saving log entries.
- Uploading logs to an API endpoint.

---

## Target User

- The site owner or authorized IdeaDump user debugging raw API logs from application integrations.

## Module Scope

### Existing Route

Use the existing route:

```txt
/log-viewer
```

### Access Control

The Log Viewer belongs under the existing `logs` RBAC module.

Implementation requirements:

- add `app/log-viewer/layout.tsx`
- call `requirePageModuleAccess('logs')`
- add sidebar navigation to `/log-viewer` for users with `logs` access
- do not add `log_viewer` to `APP_MODULE_SLUGS`

### Navigation

The sidebar should expose Log Viewer near Weekly Logs. It can be a flat nav item for v1.

Suggested navigation:

```txt
Weekly Logs -> /logs
Log Viewer  -> /log-viewer
```

## Functional Requirements

### FR-001 Input Methods

The tool must support both input methods:

- paste raw log text into a textarea
- upload a `.log` or `.txt` file and read it as text in the browser

Input is processed in memory only. Clearing the page state should remove the parsed results from the current browser session.

### FR-002 Line Splitting

The parser must:

- split raw text by line breaks
- discard blank lines
- preserve the original one-based line number
- preserve the original raw line text

### FR-003 Line Classification

Each non-empty line should be classified loosely, because real labels are inconsistent.

Suggested `lineType` values:

```ts
type ParsedLogLineType = 'request' | 'response' | 'content_data' | 'other';
```

Classification rules:

- contains `CONTENT DATA` -> `content_data`
- contains `RESPONSE` -> `response`
- contains `REQUEST` and does not contain `RESPONSE` -> `request`
- otherwise -> `other`

Suffixes such as `ERROR3`, `SUCCESS1`, or other numeric markers must not prevent classification.

### FR-004 Timestamp Extraction

Extract a leading timestamp when present.

Supported v1 pattern:

```txt
YYYY-MM-DD HH:MM:SS.mmm
```

If the timestamp cannot be parsed, keep the line and leave the parsed timestamp empty.

### FR-005 URL And Endpoint Extraction

The parser must extract the first URL matching:

```txt
https?://...
```

The parser should also derive an endpoint grouping key from the URL path. The grouping key should ignore query strings and avoid relying on host-specific values.

Recommended grouping strategy:

- parse the URL with the standard `URL` API when possible
- remove trailing slashes
- use the last two to three meaningful path segments
- fall back to the raw URL if parsing fails

Example:

```txt
https://example.com/yesshop-admin/api/user/getUserData?x=1
```

Endpoint key:

```txt
/api/user/getUserData
```

### FR-006 Payload Extraction

Payload extraction must not rely on a greedy JSON regex.

The parser should:

- find the first `{` or `[`
- use bracket matching to locate the matching closing `}` or `]`
- correctly handle nested objects and arrays
- attempt `JSON.parse()` on the extracted substring
- store raw payload text if parsing fails
- set `payloadParseError: true` when JSON extraction succeeds but parsing fails

### FR-007 Status Code Extraction

Extract status codes from patterns such as:

```txt
(200)
(400)
(500)
```

Status codes should be stored on parsed response lines when present.

### FR-008 Content Data Handling

`CONTENT DATA >> functionName >> {...}` lines are not standalone API calls by default.

The parser/pairing layer should:

- parse `CONTENT DATA` lines as structured metadata
- extract the function name when present
- attach a content data line to the next matching request within a small lookahead window
- match by function name, endpoint key, or request payload reference when possible
- render unmatched content data as standalone unpaired content so it is not lost

### FR-009 Request/Response Pairing

Because logs do not provide a reliable shared request ID, pairing must be sequence-based per endpoint.

Required algorithm:

1. Walk parsed lines from top to bottom.
2. Maintain a `Map<endpointKey, queue[]>` of unmatched requests.
3. On a request line, push it to that endpoint queue.
4. On a response line, pop the oldest request from that endpoint queue and create a paired call.
5. If a response has no queued request, create an orphan response call.
6. At end of input, convert remaining queued requests into orphan request calls.

This should replace full-URL-only pairing when endpoint normalization is available.

### FR-010 Error Detection

A call should be flagged as an error if any request, response, or parsed payload indicates failure.

Error conditions:

- line type or raw event label contains `ERROR`
- HTTP status code exists and is outside `200-299`
- parsed payload has non-null `errorCode`
- parsed payload has non-empty `errorMessage`
- parsed payload has non-empty `displayErrorMessage`
- parsed payload has `"result": "Fail"`
- parsed payload has `responseCode` and it is not `0` or `"0"`

Orphan request/response status should be displayed separately from normal errors.

### FR-011 Search And Filtering

Search and filtering are the highest-priority v1 workflow.

The UI must include:

- free-text search
- errors-only toggle
- endpoint dropdown populated from parsed calls
- clear filters action

Search should be case-insensitive and match against:

- endpoint
- request raw line
- request payload
- response raw line
- response payload
- content data raw line
- content data payload

Filtering should operate on the paired call list, not on raw lines.

### FR-012 Rendering

The page should use the existing IdeaDump sidebar layout and design tokens.

The main content should include:

- input area at the top
- filter bar below the input area
- structured call list and detail area below filters

Each call summary should show:

- timestamp when available
- endpoint or URL
- method when available
- status code when available
- status badge for success, error, orphan request, or orphan response
- line references

Expanding or selecting a call should reveal:

- request section
- response section
- content data section when attached
- raw line preview
- parsed payload preview

### FR-013 JSON Display

Use a custom lightweight JSON renderer rather than adding a third-party dependency.

Requirements:

- object and array nodes are collapsible
- deep nodes collapse by default after depth 2
- primitive values render inline
- invalid JSON falls back to raw text
- parser errors are visible to the user

### FR-014 Unparsed Lines

Lines classified as `other` must not be silently dropped.

The UI should render a separate unparsed lines section with:

- line number
- raw text
- optional reason when known

## Data And Interface Draft

These types are planning references for `lib/logViewer/types.ts`. They should be adapted to the current implementation without adding unused exported types.

```ts
export type ParsedLogLineType = 'request' | 'response' | 'content_data' | 'other';

export type ParsedPayload = {
  raw?: string;
  json?: unknown;
  parseError: boolean;
};

export type ParsedLogLine = {
  id: string;
  rawLine: string;
  lineNumber: number;
  timestamp: string;
  timestampMs?: number;
  lineType: ParsedLogLineType;
  eventType: string;
  method?: string;
  url?: string;
  endpointKey?: string;
  statusCode?: number;
  functionName?: string;
  payload: ParsedPayload;
};

export type OrphanKind = 'request' | 'response' | null;

export type LogCall = {
  id: string;
  timestamp: string;
  endpoint: string;
  request: ParsedLogLine | null;
  response: ParsedLogLine | null;
  contentData: ParsedLogLine | null;
  isError: boolean;
  orphanKind: OrphanKind;
  lineRefs: number[];
};

export type UnparsedLogLine = {
  rawLine: string;
  lineNumber: number;
  reason?: string;
};
```

## UX Requirements

- Keep the interface dense and debugging-focused.
- Avoid decorative hero sections or marketing copy.
- Use Lucide icons only.
- Use existing atoms such as `Button`, `Card`, `Input`, and `Textarea`.
- Use design token Tailwind classes only.
- No gradients or shadows.
- Preserve readable monospace formatting for raw log and JSON text.
- Keep controls visible without modal workflows.

---

## Non-Functional Requirements

- Client-only parsing for v1.
- No network upload of raw log files.
- No database writes.
- No browser storage persistence.
- Full in-memory render is acceptable for v1.
- Filtering should remain responsive for typical debugging logs.
- Large-file pagination or virtualization is deferred until real usage shows the need.
- Parser utilities should be deterministic and independently testable.

## Sample Fixtures

Three real log excerpts should be committed as fixture inputs for parser verification.

### Fixture A

Large mixed log containing:

- a `device-list` request/response pair
- multi-step eKYC calls
- `getBasicAuth`
- `uploadMyKadImage`
- `getUserDataByIdentificationNo`
- credit check calls
- Jira ticket creation with binary attachment fields

Purpose:

- validate large payload handling
- validate nested JSON display
- ensure binary-looking fields do not crash parsing

### Fixture B

Two near-identical `d-dMFCheck/verification` calls in sequence sharing the same `referenceid`.

Purpose:

- validate FIFO endpoint pairing
- ensure pairing does not rely on duplicated payload fields

### Fixture C

Repeated `getUserData` request lines followed by `RESPONSE ERROR3` lines, interleaved with successful unrelated calls and `AUTH_REQUEST` / `AUTH_RESPONSE_SUCCESS` pairs.

Also includes:

```txt
CONTENT DATA >> getUserData >> {"userId":7109}
```

Purpose:

- validate error detection by label and payload
- validate FIFO pairing under interleaving
- validate content data attachment
- validate auth request/response parsing when payload is absent

## QA Checklist

- Paste input parses and renders calls.
- File upload parses and renders calls.
- Clear action removes raw input, parsed calls, selected call, and filters.
- Search matches endpoint, raw lines, request payload, response payload, and content data.
- Errors-only filter catches error labels, failing status codes, and failing payload fields.
- Endpoint dropdown filters calls correctly.
- Request/response pairs are matched FIFO by endpoint key.
- Orphan requests are labeled distinctly.
- Orphan responses are labeled distinctly.
- Content data attaches to the following matching request.
- Unmatched content data remains visible.
- Unparsed lines section shows raw text and line numbers.
- JSON parse failures do not crash the UI.
- Large nested JSON defaults to collapsed detail sections.
- `/log-viewer` redirects unauthorized users through the `logs` module guard.
- Sidebar shows Log Viewer only to users with `logs` access.
- `npm run lint` passes.
