# Film Photography Journal & Analytics Module

## Overview

This PRD defines a new journal-first module for the IdeaDump website focused on helping a single user document, browse, and reflect on their film photography workflow.

The module should act as a centralized home for:

- film roll tracking
- cost and expense logging
- processing and scanning records
- camera maintenance history
- photo archive linking
- location tagging
- analytics and reporting

The goal is to give the user one structured, visual journal for film rolls, photos, cameras, costs, and memories. Analytics should support the journal, not dominate it.

---

## Problem Statement

Film photography information is often scattered across notebooks, cloud folders, receipts, labs, and memory. That makes it difficult to answer simple but important questions such as:

- which rolls are still unused or unprocessed
- how much a roll actually cost from purchase through scanning
- which photos belong to which roll
- which cameras have been serviced and when
- how much the hobby costs over time

Without a dedicated module, the user has no reliable system for operational tracking or historical analysis.

---

## Goals

- Create a dedicated module inside the website for film photography journaling and analytics.
- Track the full lifecycle of a film roll from purchase to archive.
- Capture all major photography-related expenses in one place.
- Associate developed photos with their originating film roll.
- Provide lightweight dashboard analytics as a supporting sub-module.
- Support cloud-folder-based photo linking for a future photo sync workflow.

---

## Non-Goals

- RAW editing or image manipulation
- social sharing features
- marketplace or community features
- multi-user collaboration
- print ordering
- advanced AI photo editing

---

## Target User

- Single primary user: the site owner managing their own film photography records.

## Functional Requirements

### FR-001 Film Roll Records

The system should allow the user to create, edit, view, and delete film roll records.

Suggested fields:

| Field | Type | Required |
|-------|------|----------|
| Film Name | string | Yes |
| Brand | string | Yes |
| Format | enum | Yes |
| ISO | number | Yes |
| Purchase Price | decimal | No |
| Camera | relation or string | No |
| Location | relation or string | No |
| Notes | text | No |

Accepted formats for v1:

- 35mm
- 120
- Large Format

### FR-002 Film Roll Lifecycle

Each film roll should support status tracking through its lifecycle.

Suggested statuses:

```text
UNUSED
LOADED
SHOOTING
AWAITING_PROCESSING
PROCESSING
PROCESSED
ARCHIVED
```

### FR-003 Film Cupboard Display

The film cupboard should be the main journal browsing experience. It should feel like opening a cupboard or storage shelf of film boxes.

Each film roll should be represented as a box or slot inside the cupboard-style display. The visual treatment should make the archive feel personal and collectible, while still keeping the information easy to scan.

Each film box should show key roll information, such as:

- film name
- brand
- format
- ISO
- status
- cover image or visual marker, if available

Pressing a film roll box should open that roll's detail view.

The cupboard display should support future growth as more rolls are added. It should still work when the user has only a few rolls and should not feel empty or broken with a small collection.

### FR-004 Photobook Roll Detail View

When a user opens a film roll, the detail view should feel like a photobook for that roll. This should be the main journal reading and browsing experience.

The photobook view should combine:

- roll metadata
- camera used
- roll status
- cost summary
- processing details
- photos from the roll
- favorite shots
- notes or journal text

The main interaction should emphasize browsing the memories from that roll, not only editing fields. Editing can exist, but the default feeling should be archival and visual.

### FR-005 Processing Records

The system should allow one or more processing records to be attached to a film roll.

Suggested fields:

| Field | Type |
|-------|------|
| Lab Name | string |
| Processing Cost | decimal |
| Scanning Cost | decimal |
| Shipping Cost | decimal |
| Processing Date | date |

### FR-006 Camera Maintenance Records

The system should allow maintenance activities to be recorded for cameras.

Supported activity examples:

- CLA
- Lens Cleaning
- Light Seal Replacement
- Repair
- Battery Replacement
- Custom Maintenance

### FR-007 Expense Calculation

The system should automatically calculate total roll cost and overall hobby cost.

Suggested formulas:

```text
Total Roll Cost
= film purchase cost
+ processing cost
+ scanning cost
+ shipping cost
```

```text
Total Hobby Cost
= sum(all roll costs)
+ sum(all maintenance costs)
```

### FR-008 Cost Efficiency Metrics

The system should support cost-per-frame and cost-per-successful-photo calculations.

Suggested fields:

| Field | Type |
|-------|------|
| Total Frames | number |
| Successful Photos | number |

Suggested formulas:

```text
Cost Per Frame = Total Roll Cost / Total Frames
```

```text
Cost Per Successful Photo = Total Roll Cost / Successful Photos
```

### FR-009 Location Tagging

The system should support attaching location metadata to film rolls.

Suggested location data:

- Location Name
- Latitude
- Longitude

Potential later enhancement:

- reverse geocoding
- map display

### FR-010 Photo Archive Association

The system should allow a film roll to be associated with a set of photos or a gallery reference.

Suggested metadata:

| Field | Type |
|-------|------|
| Cover Image | URL |
| Gallery URL | URL |
| Photo Count | number |

The module should also support:

- favorite photos
- thumbnail display
- linking photos back to a roll detail view

### FR-011 Google Drive Integration

The system should support a future Google Drive integration for photo syncing.

Planned capabilities:

- connect Google Drive account
- choose folders
- read image metadata
- generate thumbnails
- sync folder contents

Suggested OAuth scope:

```text
drive.readonly
```

Supported image formats for v1 planning:

- JPG
- JPEG
- PNG
- WEBP

### FR-012 Dashboard Summary Sub-Module

The dashboard should be a supporting sub-module, not the main experience. It should give the user a fast summary of collection size, shooting output, and money spent, while the journal/cupboard experience remains the core of the module.

Must-have dashboard metrics for v1:

- total pictures taken
- total money spent on film photography
- total cameras owned
- total film rolls tracked

The camera count should be treated as a growing collection metric. Even though the user currently owns one camera, the system should support adding more cameras over time without changing the dashboard model.

Suggested financial metrics:

- total spending
- monthly spending
- annual spending
- maintenance cost
- average spend per roll

Suggested film metrics:

- total rolls
- processed rolls
- unprocessed rolls
- rolls currently loaded or shooting

Suggested photography metrics:

- total photos
- successful photos
- average cost per photo

Suggested camera metrics:

- total cameras
- most used camera
- latest camera added
- cameras with maintenance records

## Non-Functional Requirements

- responsive UI for desktop and mobile use
- scalable enough for long-term personal archive growth
- fast dashboard loading
- secure auth and token storage if integrations are enabled
- background sync architecture for cloud photo syncing
- maintainable data model that can expand later

## Open Planning Questions

These should be resolved before implementation starts:

1. Should this module be a standalone top-level section or attached to an existing project inside IdeaDump?
2. Should cameras be reusable entities from day one so the dashboard can show camera collection growth?
3. Should photos be stored directly by the app, or only linked from cloud storage in v1?
4. Which dashboard metrics are must-have for the first release versus phase 2?
5. Should Google Drive integration ship in v1, or be designed now and deferred to a later phase?
6. Should total pictures taken count all frames, only scanned photos, or only selected successful photos?
7. Should the film cupboard use realistic film box visuals, simplified illustrated boxes, or plain data cards styled like storage slots?
8. Should the photobook detail view use page-turning interaction, a simple gallery layout, or tabs for photos, costs, and notes?
