# Log Viewer Module (Client-only)

## Overview

This PRD proposes a new **Log Viewer** module inside the existing IdeaDump app.

The module targets a specific pain point: application/network logs are currently written as **single-line text entries**, which makes it hard to:

- visually parse request/response details
- inspect JSON payloads
- find errors quickly
- understand which response belongs to which request

This Log Viewer is **client-only** (for personal use): the user uploads a `.txt` log file in the browser, and all parsing/formatting happens locally.

---

## Problem Statement

**Current**: logs are a long list of single-line entries. JSON payloads are not formatted. Request/response are not grouped.

**Impact**: slow debugging, easy to miss errors, difficult to search by endpoint or payload content.

---

## Goals

1. Upload a `.txt` log file and render it as a structured, readable view.
2. Parse each log line into structured fields (timestamp, type, method, URL, status, payload).
3. Pretty-print JSON payloads (request/response) and keep raw text accessible.
4. Automatically group/pair request + response(s) using **exact URL** matching.
5. Provide quick search + filters to find errors and specific calls fast.

---

## Non-Goals (Out of Scope)

- Server-side log ingestion or reading logs from disk/remote servers.
- Persisting uploaded logs to database (Supabase) or sharing logs.
- Cross-file correlation (pairing across multiple uploaded files).
- Normalizing or templating URLs for pairing (pairing key is **exact URL** only).

---

## Target User

- Single user (you), using the app as a personal debugging utility.

---

## Assumptions / Constraints

- **Client-only** processing.
- Log files are typically **a few MB** (should remain responsive).
- Log lines are generally time-ordered (best-effort pairing relies on order).
- Not all requests have a `guid` (do not rely on it for pairing).
- A single request **may have multiple response lines**.

---

## Log Line Formats (Observed)

Example request:

```
2026-03-16 16:53:12.243 REQUEST  >>>>>>> POST  >>>>>> URL: https://example/api/foo >>>>>> {"a":1}
```

Example response:

```
2026-03-16 16:53:13.688 RESPONSE_SUCCESS  >>>>>>> (200)  >>>>>> https://example/api/foo >>>>>> {"status":true}
```

Example response with no body:

```
2026-03-16 16:53:13.620 RESPONSE_SUCCESS  >>>>>>> (204)  >>>>>> https://example/api/bar/123 >>>>>> (No Body)
```

---

## Supported Event Types

Response-related types (hardcoded on app side):

```kotlin
const val RESPONSE = "RESPONSE"
const val REQUEST = "REQUEST"
const val RESPONSE_ERROR = "RESPONSE ERROR"
const val RESPONSE_ERROR1 = "RESPONSE ERROR1"
const val UNKNOWN_ERROR_OCCURRED = "Unknown error occurred"
const val AN_UNKNOWN_ERROR_OCCURRED = "An unexpected error occurred"
```

Notes:

- Logs in the wild may use minor variations such as underscores (e.g. `RESPONSE_SUCCESS`) or spacing. The parser should be tolerant and treat the **event type** as the text between the timestamp and the first `>>>>>>>` marker.
- The viewer should still display unknown event types rather than dropping them.

---

## Parsing Rules (MVP)

Each line is parsed into an event:

```ts
type LogEvent = {
  rawLine: string
  lineNumber: number
  timestamp: string // original
  timestampMs?: number // derived if parseable
  eventType: string // e.g. "REQUEST", "RESPONSE_SUCCESS", "RESPONSE ERROR"
  method?: string // REQUEST only (best-effort)
  url?: string // best-effort
  httpStatus?: number // RESPONSE only (best-effort)
  bodyKind?: "json" | "text" | "none"
  bodyRaw?: string
  bodyJson?: unknown
}
```

Extraction approach:

1. Timestamp: parse prefix `YYYY-MM-DD HH:mm:ss.SSS` (if present).
2. `eventType`: read from after timestamp until the first `>>>>>>>` marker.
3. For request lines:
   - extract method after first `>>>>>>>`
   - extract URL after `URL:` marker
4. For response lines:
   - extract status from `(###)` right after the first `>>>>>>>` marker (if present)
   - extract URL after the next `>>>>>>` marker (best-effort)
5. Body:
   - if it contains a JSON object/array substring, attempt `JSON.parse`
   - else treat as text, including `(No Body)`

If any fields fail to parse, the event remains valid and should still be rendered.

---

## Pairing / Grouping Rules (Exact URL)

### Definitions

- Pairing key: **exact URL string**, as extracted from the log line.
- A **transaction** is: `REQUEST` + `0..N` response events.

### Core Algorithm (URL FIFO)

Maintain a FIFO queue of open transactions per URL:

1. On `REQUEST` (with URL):
   - create a new transaction in the URL queue
2. On a response-type event (with URL):
   - attach it to the **oldest open** transaction for that URL
   - if no open transaction exists, mark it as an **orphan response**

### Multiple Responses per Request

Because a single request may produce multiple response lines:

- A transaction remains **open** after the first response.
- Additional response-type events with the same URL continue to attach to the same oldest open transaction.
- A transaction is considered **closed** when either:
  1) a new request with the same URL arrives *and* the current oldest open transaction already has at least one response, OR
  2) an inactivity timeout elapses (default suggestion: **10 seconds** since last attached response), OR
  3) the end of file is reached.

This produces stable groupings while remaining purely client-side and order-based.

### Confidence Heuristic

Each transaction should include a confidence label to communicate ambiguity:

- **High**: when the URL queue length was 0 → 1 (no concurrency for that URL)
- **Medium/Low**: when the URL queue length was > 1 at the time of pairing a response
- **Unknown**: missing URL on request or response line

## UX Requirements (MVP)

### Navigation

- New module/page inside the app: “Log Viewer”.

### Layout (Suggested)

- Top: upload area + quick filters
- Left: transaction list/table
- Right: details panel (request + responses)

Minimum fields to show in the list:

- Start time
- Method (if known)
- URL
- Latest response status (if any)
- Duration (request timestamp → last response timestamp, if parseable)
- Response count
- Confidence label

Details panel:

- Raw log line(s) (copyable)
- JSON viewer (pretty-printed with collapsible blocks) when JSON is present

### Search & Filters (MVP)

- Free-text search across URL + raw body text
- Filter: only show transactions with errors
  - error defined as:
    - response types `RESPONSE ERROR`, `RESPONSE ERROR1`, or
    - status code >= 400, or
    - eventType matches `Unknown error occurred` / `An unexpected error occurred`

---

## Performance Requirements

- Handle a few MB log file without freezing the UI.
- Parsing should run once on upload; subsequent filtering should be fast.
- For large lists, use list virtualization (if needed) rather than rendering thousands of rows.

---

## Security / Privacy

- Client-only: do not upload logs to server.
- Avoid storing the file contents in persistent storage by default (no localStorage / DB persistence in MVP).

---

## Acceptance Criteria

- [ ] User can upload a `.txt` file and see parsed events rendered in the UI.
- [ ] Event type parsing supports labels with spaces (e.g. `RESPONSE ERROR`) and preserves unknown types.
- [ ] JSON bodies render in pretty format and raw text remains accessible.
- [ ] Requests and responses group into transactions using **exact URL** pairing.
- [ ] A request may show multiple response lines under the same transaction.
- [ ] Orphan responses are visible and clearly labeled.
- [ ] Search works across URL + payload text.
- [ ] “Errors only” filter correctly narrows to error transactions.
- [ ] Works smoothly for log files up to at least ~5 MB.
