## Product Decision Update

The Tickets module should use **sub-pages** instead of one overloaded page.

The feature has two different user intents:

1. **Ticket creation** — used by Members, Admins, and Owners.
2. **Ticket management** — used only by Admins and Owners.

Therefore, the module should be structured as:

| Route                          | Purpose                                                   | Access               |
| ------------------------------ | --------------------------------------------------------- | -------------------- |
| `/tickets`                     | My Tickets — view tickets created by the current user     | Member, Admin, Owner |
| `/tickets/new`                 | Raise a new ticket                                        | Member, Admin, Owner |
| `/tickets/manage`              | Manage all tickets across allowed project/workspace scope | Admin, Owner only    |
| `/project/[id]` ticket section | Project-specific ticket section                           | Role-aware           |

This is better than one single page because it keeps the member experience simple while giving Admins and Owners a dedicated management workspace.

## RBAC Requirements

RBAC must be enforced **server-side**, not only in the UI.

### Member

Members can:

* Create tickets.
* View their own tickets.
* Edit their own tickets, depending on allowed status rules.
* Delete their own tickets only if the product allows this.

Members cannot:

* View all users' tickets.
* Access `/tickets/manage`.
* Update other users' tickets.
* Delete or close other users' tickets.

### Admin

Admins can:

* Create tickets.
* View their own tickets.
* Access `/tickets/manage`.
* View all tickets within their allowed project/workspace scope.
* Update ticket status, priority, source, notes, and tags.
* Close tickets.
* Delete tickets if the product allows Admin deletion.

### Owner

Owners can:

* Do everything Admins can do.
* Manage all tickets within the workspace/project scope.
* Perform destructive ticket actions if allowed by the product.

## Step 1 — Database Migration

**Create file:** `document/migrations/20260414_tickets.sql`

```sql
CREATE TABLE tickets (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  description TEXT,
  notes TEXT,
  status TEXT NOT NULL DEFAULT 'todo'
    CHECK (status IN ('todo', 'in_progress', 'to_review', 'done', 'closed')),
  priority TEXT NOT NULL DEFAULT 'medium'
    CHECK (priority IN ('low', 'medium', 'high')),
  source TEXT NOT NULL DEFAULT 'self'
    CHECK (source IN ('self', 'user_tester')),
  tags TEXT[] DEFAULT '{}',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

ALTER TABLE tickets ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view own tickets" ON tickets
  FOR SELECT USING (auth.uid() = user_id);

CREATE POLICY "Users can insert own tickets" ON tickets
  FOR INSERT WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users can update own tickets" ON tickets
  FOR UPDATE USING (auth.uid() = user_id);

CREATE POLICY "Users can delete own tickets" ON tickets
  FOR DELETE USING (auth.uid() = user_id);

CREATE OR REPLACE FUNCTION update_tickets_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tickets_updated_at
  BEFORE UPDATE ON tickets
  FOR EACH ROW EXECUTE FUNCTION update_tickets_updated_at();
```

### Important RBAC note

The initial RLS policies above only allow users to manage their own tickets. That is correct for Members, but not enough for Admin/Owner management.

Before implementing `/tickets/manage`, confirm how the current app stores user roles and project/workspace membership. Then update either:

1. RLS policies to allow Admin/Owner access, or
2. API route logic using a secure server-side Supabase client pattern already used elsewhere in the codebase.

Do not rely on frontend role checks alone.

## Step 3 — API Routes

### 3a. Create `app/api/tickets/route.ts`

Auth pattern: copy from `app/api/notes/route.ts` — use `createClient()` from `@/lib/supabase/server` and `supabase.auth.getUser()`.

Do NOT trust role values sent from the client. The API must resolve the current user's role server-side.

#### GET `/api/tickets`

Query params supported:

```txt
project_id  string UUID
status      TicketStatus
priority    'low' | 'medium' | 'high'
source      'self' | 'user_tester'
scope       'mine' | 'manage'
```

Behavior:

* Auth: `createClient()` + `supabase.auth.getUser()`.
* If no user, return `{ data: [] }`.
* Resolve the user's role server-side.
* If `scope` is missing, default to `mine`.
* If `scope = mine`:

    * Return tickets where `user_id = user.id`.
* If `scope = manage`:

    * Allow only `admin` and `owner`.
    * Return 403 for `member`.
    * Return all tickets within the user's allowed project/workspace scope.
* Apply filter params if present using `.eq()`.
* Order by `created_at DESC`.
* Return `NextResponse.json({ data })`.

#### POST `/api/tickets`

Body: `CreateTicketInput`

Behavior:

* Auth: `createClient()` + `supabase.auth.getUser()`.
* If no user, return a mock created ticket only if this is consistent with the existing notes fallback pattern.
* Validate `title` and `project_id` are present.
* Return 400 if either is missing.
* Insert into `tickets` with `user_id = user.id`.
* Return `NextResponse.json({ data }, { status: 201 })`.

### 3b. Create `app/api/tickets/[id]/route.ts`

Mirror `app/api/logs/[id]/route.ts` structure but without the `resolveIdentity` / agent API key flow.

Use the simpler `createClient()` + `getUser()` pattern.

#### PATCH `/api/tickets/[id]`

Body: `UpdateTicketInput`

Behavior:

* Auth: `createClient()` + `getUser()`.
* Return 401 if no user.
* Resolve the user's role server-side.
* Load the ticket by `id` first.
* If no ticket, return 404.
* If role is `member`, allow update only when `ticket.user_id === user.id`.
* If role is `admin` or `owner`, allow update if the ticket is inside their allowed project/workspace scope.
* Set `updated_at = new Date().toISOString()`.
* Return updated row as `{ data }`.

#### DELETE `/api/tickets/[id]`

Behavior:

* Auth same as above.
* Resolve the user's role server-side.
* Load the ticket by `id` first.
* If no ticket, return 404.
* If role is `member`, allow delete only when `ticket.user_id === user.id`, if member deletion is allowed by product policy.
* If role is `admin` or `owner`, allow delete if the ticket is inside their allowed project/workspace scope.
* Return `NextResponse.json({ success: true })`.
